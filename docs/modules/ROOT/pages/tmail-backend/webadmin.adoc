= TMail: Additional WebAdmin routes
:navtitle: Additional WebAdmin routes

The following document discusses additional webAdmin routes exposed by TMail.

It follows the conventions of xref:3.7.0@james-distributed-app:operate/webadmin.adoc[existing routes for the Apache James
Distributed Server].

== Team Mailboxes

=== Listing Team Mailboxes for a domain

....
curl -XGET http://ip:port/domains/domain.tld/team-mailboxes
....

Returns the list of Team Mailboxes for a given domain:

....
[
  {
     "name": "marketing",
     "emailAddress": "marketing@domain.tld"
  },
...
]
....

Return codes:

 - `400` Invalid domain
 - `404` Domain not found

=== Creating a Team Mailbox

....
curl -XPUT http://ip:port/domains/domain.tld/team-mailboxes/marketting
....

Allows to create a team mailbox.

Return codes:

 - `204` the team mailbox now exists (existed before or was created)
 - `400` invalid alias or name
 - `404` the domain was not found
 - `409` the team mailbox uses an email address used by another entity (user, group, or an alias)

=== Deleting a Team Mailbox

....
curl -XDELETE http://ip:port/domains/domain.tld/team-mailboxes/marketting
....

Allows to delete a team mailbox.

Return codes:

- `204` the team mailbox now no longer exists (did not exist prior the call or was effectively deleted)
- `400` invalid alias or name
- `404` the domain was not found

=== Listing members of a Team Mailbox

....
curl -XGET http://ip:port/domains/domain.tld/team-mailboxes/marketting/members
....

Returns the list of members of the `marketing@domain.tld` team mailbox:

....
[
  {
     "username": "bob@domain.tld"
     "role": "member"
  },
...
]
....

Return codes:

- `400` Invalid domain
- `404` Domain not found, or team mailbox not found

=== Adding a member to a Team Mailbox

....
curl -XPUT http://ip:port/domains/domain.tld/team-mailboxes/marketing/members/bob@domain.tld?role=member
....

Allows creating `bob@domain.tld` user as a member of `marketing@domain.tld` team mailbox.

Query paras:

- `role` Optional. Default to `member`. Value is either `manager` or `member`

Return codes:

- `204` the user is now a member of the team mailbox (either was already a member or was added)
- `400` invalid domain, team mailbox name or user name
- `404` the domain was not found, the team mailbox cannot be found

=== Removing a member for a Team Mailbox

Allows deleting `bob@domain.tld` user as a member of `marketing@domain.tld` team mailbox.

- `400` invalid domain, team mailbox name or user name

=== Listing extra ACL entries of a Team Mailbox folder

....
curl -XGET http://ip:port/domains/domain.tld/team-mailboxes/marketing/mailboxes/INBOX/extraAcl
....

Returns the extra ACL entries of the `INBOX` folder of the `marketing@domain.tld` team mailbox.
Team mailbox members and system users are excluded from the result; only entries granted outside of the standard team membership are returned.

Rights are expressed as an RFC 4314 string where each character represents a right:
`l` (Lookup), `r` (Read), `s` (WriteSeenFlag), `w` (Write), `i` (Insert), `k` (CreateMailbox),
`t` (DeleteMessages), `e` (PerformExpunge), `x` (DeleteMailbox), `a` (Administer).

....
{
  "usera@domain.tld": "lr",
  "userb@domain.tld": "lrwi"
}
....

Return codes:

- `400` invalid domain or team mailbox name
- `404` the domain was not found, the team mailbox cannot be found, or the folder does not exist

=== Setting an extra ACL entry on a Team Mailbox folder

....
curl -XPUT http://ip:port/domains/domain.tld/team-mailboxes/marketing/mailboxes/INBOX/extraAcl/usera@domain.tld \
  -d 'lr'
....

Grants the specified RFC 4314 rights to `usera@domain.tld` on the `INBOX` folder of the `marketing@domain.tld` team mailbox.
If the user already has an entry, it is replaced entirely.

This endpoint cannot be used to manage rights of existing team mailbox members or system users.

Return codes:

- `204` the rights have been set
- `400` invalid domain, team mailbox name, rights string, or the target user is a team mailbox member or a system user
- `404` the domain was not found, the team mailbox cannot be found, or the folder does not exist

=== Removing an extra ACL entry from a Team Mailbox folder

....
curl -XDELETE http://ip:port/domains/domain.tld/team-mailboxes/marketing/mailboxes/INBOX/extraAcl/usera@domain.tld
....

Removes all rights of `usera@domain.tld` on the `INBOX` folder of the `marketing@domain.tld` team mailbox.

This endpoint cannot be used to manage rights of existing team mailbox members or system users.

Return codes:

- `204` the entry no longer exists (was already absent or was removed)
- `400` invalid domain, team mailbox name, or the target user is a team mailbox member or a system user
- `404` the domain was not found, the team mailbox cannot be found, or the folder does not exist

=== Removing all extra ACL entries from a Team Mailbox folder

....
curl -XDELETE http://ip:port/domains/domain.tld/team-mailboxes/marketing/mailboxes/INBOX/extraAcl
....

Removes all extra ACL entries from the `INBOX` folder of the `marketing@domain.tld` team mailbox,
restoring it to its standard ACL (team mailbox members and system users only).

Return codes:

- `204` extra ACL entries have been cleared
- `400` invalid domain or team mailbox name
- `404` the domain was not found, the team mailbox cannot be found, or the folder does not exist

=== Getting sub-addressing status for a Team Mailbox folder

....
curl -XGET http://ip:port/domains/domain.tld/team-mailboxes/marketing/mailboxes/INBOX/subaddressing
....

Returns whether sub-addressing is enabled for the `INBOX` folder of the `marketing@domain.tld` team mailbox.
Sub-addressing is considered enabled when any sender (anonymous or authenticated) has the `Post` right on the folder,
meaning emails addressed to `marketing+INBOX@domain.tld` can be delivered directly to that folder.

Response body:

....
{
  "enabled": true
}
....

Return codes:

- `200` the sub-addressing status was retrieved successfully
- `400` invalid domain or team mailbox name
- `404` the domain was not found, the team mailbox cannot be found, or the folder does not exist

=== Setting sub-addressing status for a Team Mailbox folder

....
curl -XPUT http://ip:port/domains/domain.tld/team-mailboxes/marketing/mailboxes/INBOX/subaddressing \
  -H "Content-Type: application/json" \
  -d '{"enabled": true}'
....

Enables or disables sub-addressing for the `INBOX` folder of the `marketing@domain.tld` team mailbox.

- `enabled: true` grants the `Post` right to all senders (the `anyone` ACL entry), allowing emails sent to
  `marketing+INBOX@domain.tld` to be delivered to that folder regardless of the sender identity.
- `enabled: false` removes the `Post` right from the `anyone` ACL entry, preventing sub-addressing delivery
  to that folder.

Return codes:

- `204` the sub-addressing status was updated successfully
- `400` invalid domain, team mailbox name, or request body
- `404` the domain was not found, the team mailbox cannot be found, or the folder does not exist

=== Repositioning system rights on all Team Mailbox folders

....
curl -XPOST http://ip:port/team-mailboxes?action=repositionSystemRights
....

Ensures that the `admin` and `self` system users have full rights (`FULL_RIGHTS`) on all folders of all team mailboxes across all domains.
Intended for migration purposes. Processes all team mailboxes in a single synchronous call with no task manager wrapping.

Return codes:

- `204` system rights have been repositioned successfully
- `400` missing or invalid `action` query parameter

=== Getting the quota for a Team Mailbox

....
curl -XGET http://ip:port/domains/domain.tld/team-mailboxes/marketing/quota
....

Returns the quota details for the `marketing@domain.tld` team mailbox, including limits at each scope and current occupation:

....
{
  "global": {
    "count": 252,
    "size": 242
  },
  "domain": {
    "count": 152,
    "size": 142
  },
  "teamMailbox": {
    "count": 52,
    "size": 42
  },
  "computed": {
    "count": 52,
    "size": 42
  },
  "occupation": {
    "size": 13,
    "count": 21,
    "ratio": {
      "size": 0.25,
      "count": 0.5,
      "max": 0.5
    }
  }
}
....

- `global`: the global default quota limit
- `domain`: the domain-level default quota limit
- `teamMailbox`: the team-mailbox-specific quota limit (set at the user scope internally)
- `computed`: the effective quota limit after applying scope precedence (teamMailbox > domain > global)
- `occupation`: current usage, with `size` in bytes, `count` in messages, and `ratio` as fractions of the computed limit (0.0 when unlimited)

A field value of `{"count": null, "size": null}` means no limit is set at that scope. A value of `-1` means unlimited.

Return codes:

- `200` quota details returned
- `400` invalid domain or team mailbox name
- `404` the domain was not found or the team mailbox does not exist

=== Updating the quota for a Team Mailbox

....
curl -XPUT http://ip:port/domains/domain.tld/team-mailboxes/marketing/quota/limit
 -H "Content-Type: application/json"
 -d '{"count": 52, "size": 42}'
....

Sets the team-mailbox-specific quota limit for both count and size in a single call.
Each field can hold:

- a positive integer to set a finite limit
- `-1` to set an unlimited quota
- `null` to leave the current value unchanged

Examples:

....
{"count": 52, "size": 42}

{"count": null, "size": null}

{"count": 52, "size": -1}
....

Return codes:

- `204` the quota has been updated
- `400` invalid domain, team mailbox name, or quota value
- `404` the domain was not found or the team mailbox does not exist

== Deleted Messages Vault for Team Mailboxes

When messages are deleted from a team mailbox, they are stored in the deleted messages vault under the team mailbox's own identity (e.g. `sales@domain.tld`) rather than the internal technical owner.
This allows vault operations to target each team mailbox individually.

=== Browse deleted messages of a Team Mailbox

....
curl -XPOST 'http://ip:port/deletedMessages/users/sales@domain.tld/messages?force=true' \
  -H 'Content-Type: application/json' \
  -d '{"combinator":"and","criteria":[]}'
....

Lists deleted messages stored in the vault for the given team mailbox. The `?force=true` parameter bypasses the user existence check, which is required since `sales@domain.tld` is a virtual identity used internally by the vault rather than a real user account.

An empty query (`{"combinator":"and","criteria":[]}`) returns all deleted messages. Criteria can be added to filter by subject, sender, deletion date, etc. — see xref:3.7.0@james-distributed-app:operate/webadmin.adoc#_deleted_messages_vault[the James vault documentation] for the full query syntax.

Response:

....
[
  {
    "messageId": "...",
    "subject": "Hello",
    "sender": "alice@domain.tld",
    "recipients": ["sales@domain.tld"],
    "deletionDate": "2024-12-15T08:23:45Z",
    "deliveryDate": "2024-12-14T10:00:00Z",
    "hasAttachment": false,
    "size": 1234
  }
]
....

Return codes:

- `200` the list of deleted messages is returned
- `400` the address is invalid or the query body is malformed
- `404` no deleted messages found for this team mailbox

=== Restore deleted messages of a Team Mailbox

....
curl -XPOST 'http://ip:port/deletedMessages/teamMailbox/sales@domain.tld?action=restore'
....

Schedules a task that restores all deleted messages from the vault for the given team mailbox into its restore mailbox folder.

Response:

....
{"taskId":"5641376-02ed-47bd-bce7-b7a5110e7e98"}
....

The scheduled task has type `team-mailbox-deleted-messages-restore` and the following additional information:

....
{
  "type": "team-mailbox-deleted-messages-restore",
  "timestamp": "2024-12-15T08:23:45Z",
  "teamMailboxAddress": "sales@domain.tld",
  "successfulRestoreCount": 12,
  "errorRestoreCount": 0
}
....

Return codes:

- `201` the task has been scheduled, a `taskId` is returned
- `400` the team mailbox address is invalid
- `404` the team mailbox does not exist

== JMAP Labels

Labels are user-defined tags that can be attached to JMAP messages. Each label has a display name, an internal keyword (IMAP flag), and optional color and description. Labels can optionally be marked as read-only, which prevents them from being updated or deleted via JMAP.

=== List labels of a user

....
curl -XGET http://ip:port/users/user@domain.tld/labels
....

Returns the list of labels for the given user:

....
[
  {
    "id": "$label1",
    "displayName": "Work",
    "keyword": "$label1",
    "color": "#ff0000",
    "description": "Work emails",
    "readOnly": false
  },
  ...
]
....

The `color` and `description` fields are omitted when not set. The `readOnly` field is always present and defaults to `false`.

Return codes:

- `200` The label list was successfully retrieved
- `404` The user does not exist

=== Create a label

....
curl -XPOST http://ip:port/users/user@domain.tld/labels \
  -H "Content-Type: application/json" \
  -d '{"displayName": "Work", "color": "#ff0000", "description": "Work emails"}'
....

Creates a new label for the user. The `color`, `description`, and `keyword` fields are optional.

If `keyword` is omitted, a random value is generated. If provided, it must be both a valid JMAP keyword (RFC 8621) and a valid JMAP id (alphanumeric and hyphens only — no `$` prefix).

Returns the created label:

....
{
  "id": "$label1",
  "displayName": "Work",
  "keyword": "$label1",
  "color": "#ff0000",
  "description": "Work emails",
  "readOnly": false
}
....

Return codes:

- `201` The label was successfully created
- `400` `displayName` is missing, empty, or the `color` has an invalid format
- `404` The user does not exist

=== Update a label

....
curl -XPATCH http://ip:port/users/user@domain.tld/labels/{labelId} \
  -H "Content-Type: application/json" \
  -d '{"displayName": "Job", "color": "#abcdef", "description": null, "readOnly": true}'
....

Fully replaces `displayName`, `color`, and `description` on a label. `color` and `description` accept `null` to clear the field. The optional `readOnly` boolean field controls whether the label can be updated or deleted via JMAP — when `true`, JMAP clients receive a `forbidden` error when attempting to modify or destroy the label. Administrators can always update labels (including toggling `readOnly`) via this WebAdmin endpoint regardless of the `readOnly` value.

Return codes:

- `204` The label was successfully updated
- `400` `displayName` is missing, null, or empty, or the `color` has an invalid format
- `404` The user or the label does not exist

=== Delete a label

....
curl -XDELETE http://ip:port/users/user@domain.tld/labels/{labelId}
....

Deletes a label.

Return codes:

- `204` The label was successfully deleted (idempotent — also returned if the label did not exist)
- `404` The user does not exist

== Rate limiting

=== Update rate limits of a user

Allows to update the rate limits bound to a user.

....
curl -XPUT http://ip:port/users/{username}/ratelimits
 -H "Content-Type: application/json"
 -d '{
    "mailsSentPerMinute": 10,
    "mailsSentPerHours": 100,
    "mailsSentPerDays": 1000,
    "mailsReceivedPerMinute": 20,
    "mailsReceivedPerHours": 200,
    "mailsReceivedPerDays": 2000
}'
....

Note: null values are allowed (means no limit set)

Return codes:

- `204` The user rate limits updated successfully
- `400` Invalid request
- `404` User does not exist

=== Get rate limits of a user
....
curl -XGET http://ip:port/users/{username}/ratelimits
....

Returns the limits of a user:

....
{
    "mailsSentPerMinute": 10,
    "mailsSentPerHours": 100,
    "mailsSentPerDays": 1000,
    "mailsReceivedPerMinute": 20,
    "mailsReceivedPerHours": 200,
    "mailsReceivedPerDays": 2000
}
....

Note: Limits not set will return a null value.

Return codes:

- `200` get the rate limits of a user successfully
- `400` Invalid request
- `404` User does not exist

== Domain contacts

=== Create a contact

....
curl -XPOST http://ip:port/domains/{domain}/contacts
{
    "emailAddress": "bob@domain.tld",
    "firstname": "Bob",
    "surname": "Carpenter"
}
....

Creates a new contact attached to a domain

The `firstname` and `surname` fields are optional.

Returns the id of the created contact, as well as a Location header to communicate the URL of the created entry:

....
Location: /domains/domain.tld/contacts/bob
{
    "id": "6b427e04-11de-4674-a4e7-136986d9129e"
}
....

Return codes:

- `201` Contact created successfully
- `400` Invalid domain or mail address, or the domain and the mail address domain don't match
- `404` Domain not found

=== Update a contact

....
curl -XPUT http://ip:port/domains/{domain}/contacts/{username}
{
    "firstname": "Bobby",
    "surname": "Dupond"
}
....

Update the names of a contact. If contact did not exist, it gets created.

The `firstname` and `surname` fields are optional. If a field is omitted, it will not be updated.

Return codes:

- `204` Updated contact successfully
- `400` Invalid domain or mail address
- `404` Domain not found

=== Delete a contact

....
curl -XDELETE http://ip:port/domains/{domain}/contacts/{username}
....

Deletes a contact.

Return codes:

- `204` Deleted contact successfully
- `400` Invalid domain or mail address

=== Get a contact

....
curl -XGET http://ip:port/domains/{domain}/contacts/{username}

== Task management


=== Change a username
....
curl -XPOST http://ip:port/users/oldUser/rename/newUser?action=rename
....
Would migrate account data from `oldUser` to `newUser`.
link:https://james.apache.org/server/manage-webadmin.html#Endpoints_returning_a_task[More details about endpoints returning
a task].
Implemented migration steps from James are:

- `ForwardUsernameChangeTaskStep`: creates forward from old user to new user and migrates existing forwards
- `FilterUsernameChangeTaskStep`: migrates users filtering rules
- `DelegationUsernameChangeTaskStep`: migrates delegations where the impacted user is either delegatee or delegator

Implemented extra steps for TMail are:

- `ContactUsernameChangeTaskStep`: migrates contacts from old user to new user
- `PGPKeysUsernameChangeTaskStep`: migrates PGP public keys from old user to new user
- `LabelUsernameChangeTaskStep`: migrates JMAP labels from old to new user
- `JmapSettingsUsernameChangeTaskStep`: migrates JMAP settings from old to new user
- `RateLimitingUsernameChangeTaskStep`: migrates rate limiting from old to new user
- `SaaSAccountUsernameChangeTaskStep` (only if the SaaS module is enabled): migrates SaaS account information from old to new user

Response codes:
* 201: Success. Corresponding task id is returned.
* 400: Error in the request. Details can be found in the reported error.
The `fromStep` query parameter allows skipping previous steps, allowing to resume the username change from a failed step.
The scheduled task will have the following type `UsernameChangeTask` and the following `additionalInformation`:
....
{
        "type": "UsernameChangeTask",
        "oldUser": "jessy.jones@domain.tld",
        "newUser": "jessy.smith@domain.tld",
        "status": {
            "A": "DONE",
            "B": "FAILED",
            "C": "ABORTED"
        },
        "fromStep": null,
        "timestamp": "2023-02-17T02:54:01.246477Z"
}
....
Valid status includes:
- `SKIPPED`: bypassed via `fromStep` setting
- `WAITING`: Awaits execution
- `IN_PROGRESS`: Currently executed
- `FAILED`: Error encountered while executing this step. Check the logs.
- `ABORTED`: Won't be executed because of previous step failures.

== Mailbox and message location

=== Look up a mailbox by ID

Allows an administrator to retrieve the path of a mailbox from its internal identifier.

....
curl -XGET http://ip:port/mailboxes/{mailboxId}
....

Response body:

....
{
  "mailboxId": "15b47950-5799-11ef-91ef-0242ac120002",
  "mailboxPath": "#private:bob@domain.tld:INBOX"
}
....

- `mailboxId`: the internal identifier of the mailbox
- `mailboxPath`: the full mailbox path in the form `namespace:owner:name`

Return codes:

- `200` The mailbox was found.
- `400` The provided `mailboxId` is syntactically invalid.
- `404` No mailbox exists with that identifier.

=== Locate a message

=== Locate a message by ID

Allows an administrator to determine in which mailboxes a given message currently resides.
This is useful for auditing purposes, for example to prove what happened to a message or where it ended up.

....
curl -XGET http://ip:port/messages/{messageId}
....

Where `messageId` is the low-level identifier of the message (format depends on the backend, e.g. a plain integer for the memory backend, a UUID for Cassandra).

Response body:

....
{
  "mailboxes": [
    {
      "mailboxId": "15b47950-5799-11ef-91ef-0242ac120002",
      "mailboxPath": "#private:bob@domain.tld:INBOX"
    },
    {
      "mailboxId": "23c58061-5799-11ef-91ef-0242ac120002",
      "mailboxPath": "#private:bob@domain.tld:Sent"
    }
  ]
}
....

Each entry in `mailboxes` contains:

- `mailboxId`: the internal identifier of the mailbox (format depends on the backend)
- `mailboxPath`: the full mailbox path in the form `namespace:owner:name`

Return codes:

- `200` The message was found. The response lists all mailboxes it belongs to.
- `400` The provided `messageId` is syntactically invalid.
- `404` No mailbox contains a message with that identifier.

== JMAP OIDC

=== Backchannel logout

If you use OIDC authentication, the Backchannel logout route can be called by your OIDC provider to invalidate OIDC tokens
on TMail backend server when the user logs out of his JMAP client.

....
curl -XPOST http://ip:port/add-revoked-token?logout_token=[LOGOUT_TOKEN]
....

Where the `logout_token` is a JWT token sent by the OIDC provider during backchannel logout mechanism. It should contain
at least the `sid` that TMail will use to invalidate the token stored in cache, so that it can not be used to anymore to
access backend resources.

Response codes:

* 200: Success. Token has been invalidated.
* 400: Error in the request. Details can be found in the reported error.
* 415: Content type is invalid. It must be `application/x-www-form-urlencoded`.

== User Mails Search

This route allows administrators to search through a user's mailbox for debugging purposes
(e.g. investigating missing mail reports).

*This route must be explicitly wired at the application level — it is not active by default.*

To enable it add `com.linagora.tmail.webadmin.mailbox.UserMailsSearchRoute` in `extensions.routes` definition in `webadmin.properties`.

=== Search a user's mails

....
curl -XPOST http://ip:port/users/{username}/mails?limit={limit}&offset={offset} \
  -H 'Content-Type: application/json' \
  -d '{
    "reason": "Support ticket #4242 - user reports missing emails",
    "filter": {
      "text": "invoice",
      "from": "billing@vendor.com",
      "to": "user@domain.com",
      "subject": "Invoice",
      "hasAttachment": true,
      "hasKeywords": ["$seen"],
      "inMailboxes": ["mailboxId1"],
      "inMailboxOtherThan": ["mailboxId2"],
      "before": "2024-01-31T23:59:59Z",
      "after": "2024-01-01T00:00:00Z"
    },
    "sort": [
      { "property": "receivedAt", "isAscending": false }
    ]
  }'
....

All fields except `reason` are optional. `filter` and `sort` may be omitted entirely.

Query parameters:

- `limit` (optional, default: 50, max: 500): maximum number of results to return
- `offset` (optional, default: 0): number of results to skip

Request body fields:

|===
| Field | Type | Required | Description

| `reason`
| String
| *Yes*
| Justification for the search, recorded in the audit log.

| `filter.text`
| String
| No
| Full-text search across From, To, Cc, Bcc, subject, body, and attachments.

| `filter.from`
| String
| No
| Filter by sender address.

| `filter.to`
| String
| No
| Filter by recipient address.

| `filter.subject`
| String
| No
| Filter by subject.

| `filter.hasAttachment`
| Boolean
| No
| Filter by attachment presence.

| `filter.hasKeywords`
| String[]
| No
| Filter by flags/keywords (e.g. `$seen`, `$answered`, `$flagged`, `$draft`).

| `filter.inMailboxes`
| String[]
| No
| Restrict search to these mailbox IDs.

| `filter.inMailboxOtherThan`
| String[]
| No
| Exclude these mailbox IDs from the search.

| `filter.before`
| String (ISO-8601)
| No
| Return messages received on or before this date.

| `filter.after`
| String (ISO-8601)
| No
| Return messages received after this date.

| `sort[].property`
| String
| No
| Sort field: `receivedAt`, `from`, or `subject`.

| `sort[].isAscending`
| Boolean
| No
| Sort direction. Defaults to descending (`false`).
|===

Deleted messages are always excluded from results.

Response example:

....
[
  {
    "id": "1",
    "keywords": { "$seen": true },
    "mailboxIds": { "mailboxId1": true },
    "receivedAt": "2024-01-15T10:30:00+00:00",
    "from": [{ "name": "Alice", "email": "alice@example.com" }],
    "to": [{ "name": "Bob", "email": "bob@example.com" }],
    "hasAttachment": false,
    "preview": "Hello, please find attached...",
    "subject": "Invoice #42"
  }, ...
]
....

Return codes:

- `200` Successfully returned the list of matching mails
- `400` Missing or blank `reason`, invalid `limit`/`offset`, unknown sort property, or malformed JSON body
